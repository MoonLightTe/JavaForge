# 学习记录：Java集合框架与红黑树

## 学习目标

## 已掌握内容

### 1. Java集合框架概览
- **Collection接口**：List、Set、Queue
- **Map接口**：HashMap、TreeMap、LinkedHashMap
- **与JavaScript的对比**：理解了Java集合与JavaScript集合的对应关系

### 2. HashMap底层实现
- **核心组件**：数组（Node[] table）、链表、红黑树、哈希函数
- **哈希冲突处理**：链表法解决冲突
- **链表转红黑树条件**：
  - 链表长度超过8（TREEIFY_THRESHOLD）
  - 数组长度超过64（MIN_TREEIFY_CAPACITY）
- **关键参数**：
  - 默认初始容量：16
  - 默认负载因子：0.75
  - 树化阈值：8
  - 退化阈值：6

### 3. 红黑树基础
- **五大性质**：
  1. 节点颜色：红或黑
  2. 根节点：黑色
  3. 叶子节点(NIL)：黑色
  4. 红色节点的子节点：必须黑色
  5. 黑色节点数：从任一节点到叶子路径上的黑色节点数相同
- **性能优势**：查找、插入、删除时间复杂度均为O(log n)
- **应用场景**：TreeMap、HashMap中的链表优化

### 4. HashMap源码分析
- **put方法执行流程**：
  1. 计算键的哈希值
  2. 如果数组为空，进行初始化
  3. 根据哈希值计算数组索引
  4. 处理冲突（链表或红黑树）
  5. 检查是否需要扩容
- **负载因子0.75的权衡**：时间和空间成本的折衷

### 5. 实际应用与最佳实践
- **集合选择指南**：根据需求选择合适的集合类型
- **性能优化**：预设容量、合适的负载因子
- **使用示例**：HashMap的基本操作和遍历

## 关键洞察

### 1. 红黑树在HashMap中的作用
- 解决链表过长导致的性能退化问题
- 当链表长度超过8且数组长度超过64时转换为红黑树
- 红黑树提供稳定的O(log n)性能

### 2. 哈希冲突的处理策略
- 链表法：简单但可能退化为O(n)
- 红黑树优化：保证O(log n)但实现复杂
- Java 8的折衷方案：先链表，后红黑树

### 3. 负载因子的重要性
- 0.75是时间和空间的良好折衷
- 太小：浪费空间，频繁扩容
- 太大：哈希冲突增多，性能下降
